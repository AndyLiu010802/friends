## 一、背景与目标

把"友记"从本地个人星图升级为可以云端备份、AI 生成好友图鉴、并对单个好友进行图鉴问答的私人关系记忆系统，同时严格控制 AI token 成本（用户明确要求："我希望使用最高级 AI，但必须控制 token 成本"）。

三个重点：

1. **真正连接 Supabase 做云端备份**——本地 localStorage 仍是主数据源，Supabase 只做手动备份/恢复，不做自动双向同步。
2. **接入真实 AI 生成好友图鉴**——图鉴不再是 stub，基于 memories/likes/dislikes/notes/relationships 生成，MBTI/星座只作辅助。
3. **好友图鉴问答**——针对单个好友打字提问，AI 只基于该好友数据回答，资料不足时诚实说明。

贯穿以上三点的硬约束：**默认使用最省钱的模型，用户手动升级；调用前估算并展示成本；超过阈值需要用户确认。**

### 1.1 已核实的当前代码状态

| 检查项 | 结论 |
|---|---|
| 现有 Atlas 生成入口 | `app/api/generate-atlas/route.ts` 是纯 stub（模板字符串拼接，无 AI 调用），由 `app/atlas/[friendId]/page.tsx:29` 的 `generate()` 通过 `fetch('/api/generate-atlas', ...)` 调用 |
| 现有 `Atlas` 类型 | `lib/types.ts` 现状：`{ id, friendId, generatedAt, summary, personality, predictions, giftIdeas, warnings, rawInput }`，与本文档 §六 的新结构不同，需要整体替换（不是新增字段） |
| 现有 atlas 相关 store 函数 | `lib/store.ts` 已有 `getAtlasList()`、`saveAtlas()`、`getAtlasByFriendId()`，**没有** `deleteAtlas()`，需要新增 |
| 现有 Supabase 客户端 | `lib/supabase.ts` 已有 `pushFriend`、`deleteFriendRemote`、`pushAtlas`、`pullAll`、`uploadMedia`；`supabase` 为 `null` 时（未配置环境变量）所有函数直接 no-op return，这个"可选云同步、本地始终可用"的模式本文档延续 |
| 现有 Supabase 表结构 | `supabase-schema.sql` 已有 `friends`（`id, data jsonb, updated_at`）和 `atlas`（`id, friend_id, data jsonb, generated_at`）两张表，`data` 列均为 jsonb，允许 RLS `allow all` |
| Atlas 页面当前渲染逻辑 | 直接读 `atlas.summary/personality/predictions/giftIdeas/warnings` 四段渲染，无 confidence、无模型选择、无重新生成时的成本提示 |
| 依赖 | `package.json` 未安装 `@google/genai` 或任何 OpenAI SDK，需要新增依赖 |

结论：**不新建 `atlas_reports` 表**——现有 `atlas` 表（jsonb）已经能容纳升级后的 `Atlas` 类型（新增 `model`、`recordStats`、`evidence` 等字段），继续复用 `pushAtlas()`；只需新增 `friend_backups`（手动备份）和 `atlas_chats`（问答历史）两张表。**删除**旧 stub 路由 `app/api/generate-atlas/route.ts`，改为新路由 `app/api/ai/generate-atlas/route.ts`。

## 二、模型分级与成本控制

### 2.1 三档模型（已与用户确认的最终方案）

```ts
export type AIQualityMode = 'economy' | 'standard' | 'premium'

export const MODEL_MAP: Record<AIQualityMode, { provider: 'gemini' | 'openai'; model: string }> = {
  economy:  { provider: 'gemini', model: 'gemini-2.5-flash-lite' },
  standard: { provider: 'gemini', model: 'gemini-2.5-flash' },
  premium:  { provider: 'openai', model: 'gpt-5.5' },
}
```

UI 显示：

```text
省钱模式   （economy，默认）
标准模式   （standard）
最高级模式 （premium）
```

**默认策略（用户已确认）：全局默认 `economy`，图鉴生成和问答都一样，用户在 Atlas 页面手动切换档位**。不做"生成图鉴默认 premium"这种按场景区分默认值的规则——这是原两份文档的冲突点之一，已按用户选择的"全局默认省钱档"定稿。

不要求实现方在 economy/standard 之外额外接入 Gemini 2.5 Pro：premium 档直接对应 `gpt-5.5`，Gemini 2.5 Pro **不在** v0.4 模型清单中。

### 2.2 Premium 失败处理：不做自动静默 fallback

```text
premium (gpt-5.5) 调用失败或不可用时：
  → 不自动切换到其他模型（不引入 Claude、不引入 gemini-2.5-pro 作为 fallback）
  → 弹窗提示："最高级模型暂时不可用，是否使用标准模式重试？"
  → 用户确认后，用 standard 档（gemini-2.5-flash）重新发起同一次请求
```

原因：静默降级会让用户误以为自己用的是最高级模型却在用便宜模型。这一决定意味着 **v0.4 不需要 `ANTHROPIC_API_KEY`**，只需要 `GEMINI_API_KEY` 和 `OPENAI_API_KEY`。

### 2.3 默认档位规则细化

```text
生成完整图鉴：默认 economy（用户可升级到 standard/premium）
图鉴问答：默认 economy（用户可升级）
```

Atlas 页面的模型选择器（下拉/分段控件）同时控制"生成图鉴"和"问答"两个功能，选择后记忆在页面 state 中（不做跨会话持久化，每次进入 Atlas 页面重置为 economy）。

### 2.4 Token 估算工具

新增 `lib/ai/tokenEstimate.ts`：

```ts
export interface TokenEstimate {
  estimatedInputTokens: number
  estimatedOutputTokens: number
  estimatedCostUsd: number
  level: 'low' | 'medium' | 'high' | 'very-high'
}

export function estimateTextTokens(text: string): number {
  const chineseChars = (text.match(/[一-鿿]/g) ?? []).length
  const otherChars = text.length - chineseChars
  return Math.ceil(chineseChars / 1.5 + otherChars / 4)
}

export function estimateAtlasGenerationCost(
  context: FriendAtlasContext,
  mode: AIQualityMode
): TokenEstimate

export function estimateAtlasQuestionCost(input: {
  context: FriendAtlasContext
  atlas?: Atlas
  recentMessages: AtlasChatMessage[]
  question: string
  mode: AIQualityMode
}): TokenEstimate
```

- 输入 token 数：对拼好的完整 prompt 文本跑 `estimateTextTokens`，再加 20% buffer：`Math.ceil(tokens * 1.2)`。
- 输出 token 数：直接取该 `mode` + 场景对应的 `OUTPUT_LIMITS`（见 §2.6）作为估算上限。
- 成本单价表（写死在 `tokenEstimate.ts` 顶部常量，按官方定价）：

```ts
const PRICING_USD_PER_1M: Record<string, { input: number; output: number }> = {
  'gemini-2.5-flash-lite': { input: 0.10, output: 0.40 },
  'gemini-2.5-flash':      { input: 0.30, output: 2.50 },
  'gpt-5.5':                { input: 5.00, output: 15.00 }, // 占位值，实现时以 OpenAI 官方最新定价为准
}
```

  `gpt-5.5` 的定价在实现阶段需要实现者核实 OpenAI 官方最新定价并替换占位值（写代码时不要假设上面的数字是准的，OpenAI 定价变动频繁）。

- `level` 分档（按 `estimatedCostUsd`）：`< $0.1` → `low`，`< $0.5` → `medium`，`< $1` → `high`，`>= $1` → `very-high`。

### 2.5 调用前成本提示与确认弹窗

Atlas 页面生成图鉴按钮点击前（不是点击后才算）显示：

```text
预计本次会消耗约 6,800 input tokens + 1,500 output tokens。
最高级模式预计成本约 $0.47。
```

阈值规则：

```text
预计成本 > $1  → 显示确认弹窗："这次图鉴使用的记录较多，预计成本超过 $1。是否继续？"（用户可取消或继续）
预计成本 > $5  → 强制建议切换："这次请求成本较高。建议先使用标准模式生成，或者减少传入 memories 数量。"（阻断式提示，不能直接继续发起 premium 请求，需用户明确选择"仍然继续"或"切换到标准模式"）
```

问答（AtlasChatBox）同样在发送前跑 `estimateAtlasQuestionCost`，超过 $1 用同样的确认弹窗（问答场景实际很难触达 $5，但代码逻辑复用同一套阈值判断函数）。

### 2.6 输出长度限制

```ts
export const OUTPUT_LIMITS: Record<'atlas' | 'question', Record<AIQualityMode, number>> = {
  atlas:    { economy: 1200, standard: 1600, premium: 2200 },
  question: { economy: 500,  standard: 800,  premium: 1200 },
}
```

调用 AI 时把对应值传入 `GenerateOptions.maxOutputTokens`（见 §八 Provider 封装）。

### 2.7 问答时按问题筛选 memories（不传全部）

新增 `lib/ai/selectMemoriesForQuestion.ts` 导出：

```ts
export function selectRelevantMemoriesForQuestion(friend: Friend, question: string): Memory[]
```

规则（按优先级累加，去重后不超过 15 条）：

1. 总是包含最近 5 条 memories（按 `date` 降序）。
2. 问题包含「礼物 / 生日 / 喜欢 / 买 / 送」→ 优先包含 `tags` 或 `content` 中含有「礼物、生日、喜欢、想要、买、讨厌」的 memories。
3. 问题包含「聊 / 话题 / 见面」→ 优先包含最近的 memories（复用规则 1 的最近集合，不重复取）。
4. 问题包含「关系 / 变近 / 变远 / 为什么」→ 包含较大时间跨度：最早 3 条 + 最近 8 条。
5. 最多传 15 条 memories（按上述规则合并后如果超过 15 条，按 `date` 降序截断保留前 15）。
6. 单条 `memory.content` 超过 600 字时截断（保留前 600 字 + `"……"`）。

### 2.8 生成图鉴时限制 memories

新增 `lib/ai/selectMemoriesForAtlas.ts` 导出：

```ts
export function selectMemoriesForAtlas(friend: Friend): Memory[]
```

规则（各来源取出后按 `id` 去重，最多保留 30 条）：

1. 最近 15 条（按 `date` 降序）。
2. 最早 3 条（按 `date` 升序）。
3. `content` 最长的 5 条。
4. `tags` 数量最多的 5 条。
5. `content` 或 `tags` 中包含关键词「生日、礼物、吵架、重要、喜欢、不喜欢」的 memories。

去重后若仍超过 30 条，按 `date` 降序截断保留前 30。若好友总 memories 数超过 30，在 prompt 中额外注明：

```text
该好友共有 {total} 条 memories，本次选取了最有代表性的 30 条。
```

（这句话由 `buildAtlasPrompt` 在检测到 `friend.memories.length > 30` 时拼接，不是 `selectMemoriesForAtlas` 自己输出。）

## 四、Supabase 云端备份设计

### 4.1 架构原则

localStorage 仍是主数据源。Supabase 只做 Backup / Restore / Atlas backup（复用现有 `atlas` 表）/ AI chat backup。不做自动双向同步（避免本地覆盖云端、云端覆盖本地、多设备冲突、旧数据覆盖新数据）。

用户操作只有两个明确按钮：**备份到云端** / **从云端恢复**。

### 4.2 环境变量

新增/更新 `.env.local.example`：

```env
NEXT_PUBLIC_SUPABASE_URL=你的 Supabase URL
NEXT_PUBLIC_SUPABASE_ANON_KEY=你的 Supabase anon key

GEMINI_API_KEY=你的 Gemini API key
GEMINI_DEFAULT_MODEL=gemini-2.5-flash-lite

OPENAI_API_KEY=你的 OpenAI API key
```

`GEMINI_API_KEY` 和 `OPENAI_API_KEY` 绝对不能加 `NEXT_PUBLIC_` 前缀、绝对不能在 client component 里 import 调用它们的模块——所有 AI 调用必须走 Next.js API route。

### 4.3 Supabase SQL 变更

在 `supabase-schema.sql` 末尾追加（不改动已有的 `friends`、`atlas` 表）：

```sql
create table if not exists friend_backups (
  id text primary key,
  backup_name text not null,
  friends jsonb not null,
  atlas_list jsonb not null default '[]'::jsonb,
  ai_chats jsonb not null default '[]'::jsonb,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create table if not exists atlas_chats (
  id text primary key,
  friend_id text not null,
  messages jsonb not null default '[]'::jsonb,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

alter table friend_backups enable row level security;
alter table atlas_chats enable row level security;

drop policy if exists "allow all friend_backups" on friend_backups;
drop policy if exists "allow all atlas_chats" on atlas_chats;

create policy "allow all friend_backups" on friend_backups for all using (true) with check (true);
create policy "allow all atlas_chats" on atlas_chats for all using (true) with check (true);
```

当前只是个人软件，RLS 先 allow all；未来如果加入登录系统，必须改成按 `user_id` 限制访问。

### 5.3 备份到云端流程

点击"备份到云端" → 确认弹窗：

```text
这会把当前本地数据备份到 Supabase。不会删除本地数据，也不会自动合并云端数据。
```

确认后：

```ts
const payload: CloudBackupPayload = {
  id: crypto.randomUUID(),
  backupName: `友记备份 ${new Date().toLocaleString()}`,
  friends: getFriends(),
  atlasList: getAtlasList(),
  aiChats: getAtlasChats(),
  createdAt: new Date().toISOString(),
  updatedAt: new Date().toISOString(),
}
await backupToCloud(payload)
```

完成后提示：`备份完成：已保存 X 位朋友、Y 份图鉴、Z 条问答记录。`

### 5.4 从云端恢复流程

用户从 `listCloudBackups()` 渲染的列表中选择一条 → 点击"从此备份恢复" → 确认弹窗：

```text
这会用云端备份覆盖当前本地数据。当前 localStorage 中的数据会被替换。建议你先备份当前本地数据。
```

确认后：

```ts
const payload = await restoreFromCloud(backupId)
if (payload) {
  replaceFriends(payload.friends)
  replaceAtlasList(payload.atlasList)
  replaceAtlasChats(payload.aiChats)
}
```

`lib/store.ts` 新增：

```ts
export function replaceFriends(friends: Friend[]): void
export function replaceAtlasList(atlasList: Atlas[]): void
export function replaceAtlasChats(chats: AtlasChat[]): void
```

三个函数直接整体覆盖对应的 localStorage key（不做逐条 merge）。

## 六、AI 图鉴数据设计

### 6.1 `Atlas` 类型升级（替换现有定义，非新增字段）

```ts
export interface Atlas {
  id: string
  friendId: string
  generatedAt: string
  model: string

  recordStats: {
    memoryCount: number
    relationshipCount: number
    likesCount: number
    dislikesCount: number
    hobbiesCount: number
    noteLength: number
    confidence: 'low' | 'medium' | 'high'
  }

  summary: string
  roleInMyLife: string
  keyDetailsToRemember: string[]
  recentInteractionInsight: string
  conversationTopics: string[]
  giftIdeas: string[]
  warnings: string[]
  suitableActivities: string[]
  relationshipTrend: string

  evidence: AtlasEvidence[]
  rawInput: Partial<Friend>
}

export interface AtlasEvidence {
  type: 'memory' | 'like' | 'dislike' | 'hobby' | 'note' | 'relationship'
  id?: string
  date?: string
  text: string
}
```

旧字段 `personality`、`predictions` 被 `roleInMyLife`、`recentInteractionInsight`、`relationshipTrend` 等替代——旧的 stub 生成的 Atlas 记录在 v0.4 上线后视为过期数据，`getAtlasByFriendId` 读到旧结构时展示层按新结构的可选字段处理（旧记录会被判定为"字段缺失"，Atlas 页面据此提示用户"图鉴格式已更新，请重新生成"）。

### 6.2 Confidence 计算

新增 `lib/atlasConfidence.ts`：

```ts
export function calculateAtlasConfidence(friend: Friend): {
  score: number
  level: 'low' | 'medium' | 'high'
  reason: string
}
```

规则：

```text
memoryCount >= 8 且 notes/likes/hobbies 较完整 → high
memoryCount >= 3 或资料较完整 → medium
只有名字/少量字段 → low
```

`reason` 示例：

```text
low:    资料还比较少，图鉴只能给出初步印象。
medium: 已有一些回忆和喜好记录，图鉴可以做出较有参考价值的分析。
high:   已有较多回忆、喜好和关系记录，图鉴分析会更贴近真实相处情况。
```

## 七、AI Context Builder

新增 `lib/ai/contextBuilder.ts`：

```ts
export function buildFriendAtlasContext(
  friend: Friend,
  allFriends: Friend[],
  question?: string
): FriendAtlasContext
```

- 未传 `question`（图鉴生成场景）→ 内部调用 `selectMemoriesForAtlas(friend)`。
- 传了 `question`（问答场景）→ 内部调用 `selectRelevantMemoriesForQuestion(friend, question)`。

```ts
export interface FriendAtlasContext {
  friend: {
    id: string
    name: string
    nickname?: string
    birthday?: string
    zodiac?: string
    mbti?: string
    important: boolean
    notes?: string
  }
  likes: string[]
  dislikes: string[]
  hobbies: string[]
  memories: { id: string; date: string; title: string; content: string; tags: string[] }[]
  relationships: { friendId: string; friendName: string; label: string; closeness: 1 | 2 | 3 }[]
  stats: {
    memoryCount: number
    relationshipCount: number
    profileCompletion: number
    growthStage: string
    energyLevel: string
    confidence: 'low' | 'medium' | 'high'

}
```

`relationships[].friendName` 通过 `allFriends` 查找 `relationship.friendId` 对应的名字（找不到则回填 `'（已删除的好友）'`）。

不把完整 `Media` URL 传给 AI；v0.4 暂时不分析图片或视频，只传文本记录（`memories` 数组不包含 `media` 字段）。`stats.profileCompletion`/`growthStage`/`energyLevel` 复用 v0.2 已有的 `lib/profileCompletion.ts`/`lib/growthStage.ts`/`lib/friendEnergy.ts`，`stats.confidence` 复用 §6.2 的 `calculateAtlasConfidence`。

## 八、AI Provider 封装

新增 `lib/ai/provider.ts`：

```ts
export type AIProvider = 'gemini' | 'openai'

export interface GenerateOptions {
  model: string
  temperature?: number
  maxOutputTokens: number
}

export async function generateWithAI(
  provider: AIProvider,
  prompt: string,

```

内部按 `provider` 分发到 `lib/ai/gemini.ts` 或 `lib/ai/openai.ts`（各自封装官方 SDK 调用，返回纯文本）。不在客户端调用；必须通过 API route：`app/api/ai/generate-atlas/route.ts`、`app/api/ai/ask-atlas/route.ts`。

### 8.1 `lib/ai/gemini.ts`

```ts
import { GoogleGenAI } from '@google/genai'

const ai = new GoogleGenAI({ apiKey: process.env.GEMINI_API_KEY! })

export async function callGemini(prompt: string, model: string, maxOutputTokens: number): Promise<string> {
  const response = await ai.models.generateContent({
    model,
    contents: prompt,
    config: { maxOutputTokens },
  })
  return response.text ?? ''
}
```

依赖：`npm install @google/genai`。

### 8.2 `lib/ai/openai.ts`

```ts
import OpenAI from 'openai'

const client = new OpenAI({ apiKey: process.env.OPENAI_API_KEY! })

export async function callOpenAI(prompt: string, model: string, maxOutputTokens: number): Promise<string> {
  const response = await client.chat.completions.create({
    model,
    messages: [{ role: 'user', content: prompt }],
    max_completion_tokens: maxOutputTokens,
  })
  return response.choices[0]?.message?.content ?? ''
}
```

依赖：`npm install openai`。实现时以 OpenAI 官方 SDK 当前版本的实际 API 形状为准（字段名如有变动以 SDK 类型定义为准，上面是设计意图，不是逐字契约）。

`generateWithAI('gemini', ...)` 调 `callGemini`；`generateWithAI('openai', ...)` 调 `callOpenAI`。`GEMINI_API_KEY`/`OPENAI_API_KEY` 缺失时，对应函数在调用时抛出明确错误（`缺少 GEMINI_API_KEY，请在 .env.local 中配置。` / 同理 OpenAI），由 API route 捕获后转成 §十四 的错误响应，不在模块加载时就抛错（避免没配置其中一个 key 时另一个 provider 也用不了）。

## 九、AI API Route

### 9.1 生成图鉴：`app/api/ai/generate-atlas/route.ts`

请求体：

```ts
interface GenerateAtlasRequest {
  context: FriendAtlasContext
  mode: AIQualityMode   // 'economy' | 'standard' | 'premium'
}
```

流程：

```text
1. 客户端读取 friend + allFriends（localStorage 只在浏览器可访问，API route 拿不到）
2. 客户端调用 buildFriendAtlasContext(friend, allFriends) 得到 context
3. 客户端把 { context, mode } POST 给这个 route
4. route 内部：MODEL_MAP[mode] 取 provider+model → buildAtlasPrompt(context) → generateWithAI(provider, prompt, { model, maxOutputTokens: OUTPUT_LIMITS.atlas[mode] })
5. safeParseAIJson 解析返回文本
6. 解析成功 → 拼装完整 Atlas 对象（补上 id/friendId/generatedAt/model/recordStats）返回
7. 解析失败 → 返回 { ok: false, raw, error } 结构，由前端展示"AI 返回格式不完整"提示（不是 500，是 200 + ok:false，方便前端展示原始文本兜底）
```

`recordStats` 由 route 端根据传入的 `context.stats`/`context` 里的计数直接计算填充（不依赖 AI 输出这部分）。

### 9.2 问答：`app/api/ai/ask-atlas/route.ts`

请求体：

```ts
interface AskAtlasRequest {
  context: FriendAtlasContext   // 已经是针对该问题筛选过 memories 的 context（客户端传 question 给 buildFriendAtlasContext 生成）
  atlas?: Atlas
  messages: AtlasChatMessage[]  // 只传最近 8 条（客户端截断后传入）
  question: string
  mode: AIQualityMode
}
```

响应：

```ts
interface AskAtlasResponse {
  ok: boolean
  answer?: string
  evidence?: AtlasEvidence[]
  suggestedFollowUps?: string[]
  raw?: string
  error?: string
}
```

流程与 9.1 类似：`buildAtlasQuestionPrompt(...)` → `generateWithAI(...)` → `safeParseAIJson` → 返回。
